## 개요

기존에 제출한 공시의 내용을 수정하여 다시 제출하는 공시. 기존 공시의 rcept_no가 무효화되고 새로운 rcept_no가 부여됨.

## 소스

- **제출처**: [[DART-OpenAPI]]
- **의무 여부**: 의무 (오류 발견 시)
- **공시 주기**: 수시 (기존 공시 수정 시)

## 주요 내용

- 정정 대상 원본 공시 정보
- 정정 전/후 내용 비교
- 정정 사유
- 정정 후 전체 문서

## 전체 문서 구조 (트리)

```
정정공시
├── 원본 공시 정보
│   ├── 원본 rcept_no
│   └── 원본 공시 유형
├── 정정 내용
│   ├── 정정 항목
│   ├── 정정 전 내용
│   └── 정정 후 내용
└── 정정 사유
```

API: DART pblntf_ty="C" (정정) 또는 원본 rcept_no로 조회 시 정정본 있으면 별도 flagging 필요
주의: 소집공고 정정 시 후보자 변경 가능 → 시간순서 규칙 적용 필요

## API 필드 vs 원문 대응

| API 필드 / 방법 | 의미 | 주의사항 |
|-----------------|------|----------|
| `pblntf_ty="C"` 검색 | 정정공시 목록 조회 | 원본 공시 유형과 무관하게 C로 분류됨 |
| `rcept_no` (정정본) | 새 접수번호 | 원본 rcept_no와 다름 — 원본은 무효화 |
| `rcp_no` (원본, 문서 내) | 원본 접수번호 참조 | 문서 내 프리앰블에서 파싱 |
| `_strip_correction_preamble` | OPM 파서 함수 | 정정 머리말 제거 후 실제 공시 내용 파싱 |

**정정 후에는 항상 최신 rcept_no 사용** — 원본 rcept_no로 내용 조회 시 구 버전이 반환될 수 있음.

## 파싱 주의사항

- 소집공고 정정 시 후보자 변경·추가·삭제 가능 — [[시간순서-규칙]] 상 정정본이 항상 우선
- 동일 공시가 여러 번 정정될 수 있음 — 가장 최근 rcept_no만 유효
- `agm_corrections` tool의 `_strip_correction_preamble`으로 정정 머리말 제거 후 원본 파서에 재투입
- 정정공시를 원본으로 다시 정정하는 중첩 정정 사례 존재 — rcept_no 체인 추적 필요

## OPM에서의 활용

agm_corrections tool이 정정공시를 파싱:
- [[DART-OpenAPI]] search_filings에서 정정공고 검색
- rcept_no 변경 추적 (기존 번호 무효화)
- 파서의 _strip_correction_preamble 함수로 정정 프리앰블 제거 후 파싱
- [[agm-tool-rule]]에서 정정공고 중복 파싱 방지 규칙 정의

## 관련

[[주주총회소집공고]]의 정정이 가장 빈번. [[시간순서-규칙]]에 따라 정정 시점이 원본보다 후이므로, 최신 rcept_no를 사용해야 함. [[v4-스키마]]에서 rceptNo 필드는 최신 정정본 기준.
