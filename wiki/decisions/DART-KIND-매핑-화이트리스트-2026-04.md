# DART-KIND 매핑 화이트리스트 (2026-04)

## 목적

`open-proxy-mcp`가 현재 사용하는 공시 중에서, 어떤 공시는 `DART -> KIND`를 안전하게 병행 조회할 수 있고, 어떤 공시는 `DART 단일 소스`로 남겨야 하는지 정리한다.

이번 점검은 `PDF 다운로드를 기본 경로에서 제외`한다는 현재 방침을 전제로 했다.

즉, 운영 원칙은 아래와 같다.

- 기본 축은 `DART API + DART document.xml`
- 필요 시 `KIND HTML` 병행
- `Naver`는 참고 보조
- `PDF`는 기본 경로에서 제외
- KIND는 `화이트리스트 공시 유형`에만 허용

## 조사 범위

현재 레포가 실제로 사용하는 공시 유형 9개를 대상으로 1차 샘플 점검을 했다.

1차 샘플:

- 공시 유형 9개
- 유형별 회사 3개
- 총 27개 샘플

추가 검증:

- 화이트리스트 후보 공시 유형 안에서
- 다른 회사 10개 샘플 추가 조회

## 핵심 결론

1. `DART document.xml`은 1차 샘플 27개 전부 정상 조회됐다.
2. `KIND raw 번호 대입`은 위험하다.
3. `DART rcept_no 80 -> KIND acptno 00` 변환은 일부 거래소 공시에서 매우 강하게 작동했다.
4. 따라서 KIND는 `공시 유형 화이트리스트 + 제목 검증`이 붙어야 한다.
5. `화이트리스트 밖 공시`는 KIND를 붙이지 말고 `DART-only`로 가는 것이 안전하다.

## False Match 예시

`RAW_OK`가 곧 `매핑 성공`은 아니었다.

- 고려아연 `주주총회소집공고` raw KIND 조회
  - 실제 KIND 제목: `매출액 또는 손익구조 30% 이상 변동`
- 영풍 `의결권대리행사권유참고서류` raw KIND 조회
  - 실제 KIND 제목: `감사보고서 제출`
- 한진칼 `대량보유 상황보고` raw KIND 조회
  - 실제 KIND 제목: `기업설명회(IR) 개최(안내공시)`
- 삼성전자 `자기주식 이벤트` raw KIND 조회
  - 실제 KIND 본문: 다른 회사 `포메탈` 사업보고서

따라서 `번호가 KIND에서 열린다`만으로는 안 되고, 최소한 아래 3개가 맞아야 한다.

- 회사명
- 공시명
- 접수일

## KIND 화이트리스트

아래 공시만 KIND 병행을 허용한다.

| whitelist_key | 공시 유형 | DART 검색 기준 | KIND 매핑 규칙 | 제목 검증 |
|---|---|---|---|---|
| `agm_result` | 주주총회결과 | `pblntf_ty=I`, `report_nm contains 주주총회결과` | `rcept_no[8:10] == 80 -> 00 변환` | KIND 제목에 `주주총회 결과` 포함 |
| `dividend_decision` | 현금ㆍ현물배당결정 | `pblntf_ty=I`, `report_nm contains 현금/배당결정` | `80 -> 00 변환` | KIND 제목에 `현금ㆍ현물배당 결정` 포함 |
| `value_up` | 기업가치제고계획(자율공시) | `pblntf_ty=I`, `report_nm contains 기업가치제고/밸류업` | `80 -> 00 변환` | KIND 제목에 `기업가치 제고 계획` 포함 |
| `litigation_exchange_style` | 소송 등의 제기/신청/판결/결정 | `pblntf_ty=I/B`, `report_nm contains 소송/경영권분쟁소송` | `80 -> 00 변환` 우선 | KIND 제목에 `소송 등의 ...` 또는 `경영권분쟁소송` 포함 |

## 운영 규칙

### 허용

- DART에서 공시 후보를 찾는다
- 화이트리스트 공시 유형이면 KIND를 병행한다
- 단, `80 -> 00` 변환과 `제목 검증`을 같이 수행한다

### 금지

- 화이트리스트 밖 공시에 KIND를 raw 번호로 바로 넣는 것
- KIND 문서가 열렸다는 이유만으로 매핑 성공으로 간주하는 것
- 제목/회사명/날짜 대조 없이 KIND 본문을 채택하는 것

## release_v2 적용안

실제 구현에서는 아래 순서가 적합하다.

1. `DART list.json`으로 공시 유형과 `rcept_no` 확정
2. `document.xml`로 기본 본문 확보
3. 공시 유형이 화이트리스트에 있으면
4. `80 -> 00` 규칙 적용
5. KIND 제목 검증
6. 검증 통과 시에만 `source_used += kind_html`
7. 검증 실패 시 `DART-only` 유지
