# MCP 개발 Lessons Learned

OPM v1.3.0 → v2.0.0 개발 과정에서 반복적으로 나타난 결정 분기와 교훈.
2026-04-19 기준 회고.

---

## 1. Tool 수 vs 기능 — 가장 큰 분기

처음엔 "기능이 많을수록 좋다"고 생각해서 36개까지 늘렸음.
AI가 도구 수가 많을수록 무엇을 쓸지 헷갈린다는 걸 korean-law-mcp(91→15), assembly-api-mcp(15→10) 등 다른 MCP 분석에서 확인. 결국 36→11개로.

**교훈**: MCP는 "기능 완성도"보다 "AI가 올바른 tool을 선택할 확률"이 더 중요.
Tool 수를 줄이고 각 tool의 지능을 높이는 방향이 정답.

---

## 2. 데이터 소스 우선순위 — 처음부터 정해야 할 선

소스마다 품질/속도/비용이 달라서 우선순위 없이 섞으면 디버깅 지옥이 옴.

| 소스 | 품질 | 결론 |
|------|------|------|
| DART API | 빠르고 구조화 | 1순위, 분당 1000회 한도 주의 |
| DART XML | 정확하지만 느림 | 2순위 |
| KIND 크롤링 | 투표결과 유일 소스 | 화이트리스트 공시만 허용 |
| PDF | 느리고 불안정 | v2에서 기본 경로 제외 |
| 뉴스 | 참고만 가능 | 공시 절대 대체 불가 |

**교훈**: 상위 소스로 해결되면 하위 소스 접근 금지 원칙을 처음부터 명시해야 함.

---

## 3. API 필드 오해 — 원문을 직접 읽어야 한다

`ctr_stkrt` 필드를 "본인 지분율"로 잘못 이해 → KOSPI 200 전수조사에서 66개 MATCH_DIFF 발생.
실제로는 "주요계약체결(담보/신탁) 비율". 원문 트리 전수 분석으로 확인.

**교훈**: 외부 API 필드는 문서만 믿으면 안 됨. 실데이터 검증 필수.
특히 공시 도메인은 필드명이 직관적이지 않음 → comparison/ 페이지로 기록할 것.

참고: [[stkrt-vs-ctr_stkrt]]

---

## 4. 파서는 가정 금지, 트리를 먼저 봐라

"주총 공시는 이런 구조일 것이다"라고 가정하고 파서를 짜면 반드시 터짐.
KOSPI 200 199개 돌려보면 HARD_FAIL, 비표준 구조 케이스가 쏟아짐.

**교훈**: 파서 짜기 전에 실제 원문 트리를 20~30개 직접 보고 시작.
"이럴 것이다" 가정 하나씩 깨질 때마다 재작업 비용이 눈덩이처럼 커짐.

참고: [[파서-판정-등급]], [[경력-파서-벤치마크-2026-04]]

## 5. Tool과 로직 분리 — v1에서 배운 구조적 교훈

v1은 tool 안에 파싱/분석 로직이 섞여 있었음 → tool 수정 시 MCP 인터페이스까지 흔들림.
v2에서 `tools_v2/` (MCP 인터페이스) + `services/` (도메인 로직) 분리로 해결.

```
tools_v2/shareholder_meeting.py   ← 입력 검증 + 응답 포매팅만
services/shareholder_meeting.py   ← 실제 파싱/분석 로직
```

**교훈**: MCP tool은 인터페이스만. 로직은 별도 레이어 분리.
나중에 tool 이름 변경/통합 시 서비스 레이어는 그대로.

참고: [[release_v2-tool-아키텍처]]

## 6. 공개/비공개 분리 — 처음부터 결정해야 할 것

처음엔 파이프라인, 데이터, MCP 서버가 한 repo. 나중에 분리 작업이 복잡했음.

```
오픈소스 (public):  open-proxy-mcp — MCP tool + 파서 + API 클라이언트
비공개 (private):   open-proxy-ai  — 파이프라인 + 프론트엔드 + 데이터
```

**교훈**: 비즈니스 모델을 처음부터 정하고 repo 구조를 잡아야 함.
"일단 한 곳에 다 넣고 나중에 분리"는 기술 부채.

참고: [[free-paid-분리]]

---

## 7. Wiki-first — 코드보다 지식이 먼저

처음엔 코드만 짰음. "왜 이 결정을 했지?"를 6주 후에 기억 못 하는 일 반복.
wiki 도입 후 decisions/, comparison/, analysis/ 분리로 작업 속도 상승.

**교훈**: 도메인이 복잡한 MCP일수록 지식 관리가 코드 관리만큼 중요.
결정 내릴 때 이유를 기록하지 않으면 같은 실수를 반드시 반복함.

---

## 요약

MCP는 "데이터를 가져오는 것"보다 **"AI가 올바른 도구를 골라서 신뢰할 수 있는 결과를 내는 것"** 이 핵심.

그 관점에서:
- tool 수 최소화
- 소스 신뢰도 계층화
- 구조 분리 (interface / logic)
- 지식 관리 (wiki)

이 네 가지가 전부 연결되어 있음.
