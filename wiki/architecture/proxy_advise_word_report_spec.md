# proxy_advise Word 보고서 설계 스펙

## 목적

`proxy_advise` 실행 이후 사용자가 “문서화해달라”, “워드로 뽑아달라”, “보고서로 정리해달라”고 요청했을 때 사용할 **기본 Word 보고서 양식**을 설계한다.

이 스펙의 목적은 특정 의결권자문사 문서를 복제하는 것이 아니라, 다음을 동시에 만족하는 **OPM 기준 최적 Word 출력 형식**을 정하는 것이다.

- 한국 기관투자자 실무에서 읽기 쉬울 것
- 현재 OPM `proxy_advise` 데이터 구조로 최대한 자동 생성 가능할 것
- 권고, 근거, caveat, 법률/정책 레이어가 명확히 드러날 것
- 향후 Word export 기능으로 구현 가능할 정도로 구체적일 것

## 참조 입력

이 작업은 반드시 아래 두 축을 함께 본다.

### 1. 외부 형식 참조

다음 경로의 실제 샘플 문서를 우선 참조한다.

- `/Users/marcoyou/Projects/open-proxy-mcp/samples`

여기 있는 다양한 의결권자문사/기관 문서 포맷을 읽고 공통 패턴과 차이를 분류한다.

### 2. 내부 데이터/출력 제약

다음 OPM 실제 구조를 기준으로 한다.

- `proxy_advise` 관련 tool / service / renderer
- agenda / subagenda / recommendation / rationale / evidence / law layer / caveat 구조
- 현재 OPM에서 실제로 산출 가능한 필드
- 현재 없는 필드, 또는 파생 로직이 있어야만 만들 수 있는 필드

즉, 외부 샘플만 보고 예쁜 양식을 제안하는 것이 아니라, **현재 OPM 데이터로 실제 구현 가능한 Word 형식**을 설계해야 한다.

## 핵심 질문

아래 질문에 명시적으로 답해야 한다.

1. 의결권자문 보고서에서 공통적으로 반복되는 핵심 섹션은 무엇인가?
2. 샘플들 사이에서 가장 크게 다른 구조는 무엇인가?
3. OPM `proxy_advise` 데이터와 가장 자연스럽게 매핑되는 보고서 구조는 무엇인가?
4. 사용자에게 가장 먼저 보여줘야 할 정보는 무엇인가?
5. 안건별 권고와 근거는 어떤 레벨까지 본문에 두고, 어떤 내용은 부록으로 내리는 것이 좋은가?
6. 법률/정관/정책/사실관계/불확실성은 어떤 순서와 밀도로 보여주는 것이 좋은가?
7. 현재 OPM 데이터로 바로 자동 생성 가능한 섹션은 무엇인가?
8. 추가 파생 로직이나 필드가 있어야 구현 가능한 섹션은 무엇인가?
9. 최종 Word 문서는 “즉시 구현 가능한 v1”과 “이상적인 v2”를 어떻게 구분해야 하는가?

## 반드시 지킬 원칙

### 현실성

- 한국 상장사 주총/의결권 자문 문서 실무를 벗어난 추상적 템플릿을 만들지 않는다.
- 웹 UI처럼 보이는 구성을 제안하지 않는다.
- 실제 Word 문서로 배포/출력된다는 점을 전제로 설계한다.

### 데이터 정직성

- OPM에 현재 없는 필드를 있는 것처럼 가정하지 않는다.
- 없는 필드가 필요하면 “추가 필드 필요” 또는 “파생 로직 필요”로 명시한다.
- 자동 생성 가능 섹션과 사람 검토가 필요한 섹션을 분리한다.

### 제품성

- 단순 비교표나 아이디어 나열로 끝내지 않는다.
- 최종적으로 **기본 채택안 1개**를 추천해야 한다.
- 추천안은 engineering이 Word export 기능으로 구현할 수 있을 정도로 구체적이어야 한다.

### 비복제

- 특정 자문사 문서 양식을 그대로 베끼지 않는다.
- 샘플은 참고 자료이며, OPM 데이터 구조와 사용자 목적에 맞게 재구성한다.

## 권장 작업 순서

1. `samples/` 문서들을 읽고 형식 패턴을 분류한다.
2. OPM `proxy_advise` 관련 코드와 출력 구조를 읽어 실제 사용 가능한 필드를 정리한다.
3. 외부 형식 패턴과 OPM 데이터 필드 간 매핑 가능성을 비교한다.
4. Word 보고서 정보 구조를 최소 2안 이상 설계한다.
5. 각 안에 대해 readability, 실무 적합성, 자동생성 가능성, 구현 난이도, 확장성을 비교한다.
6. 기본 채택안 1개를 선택한다.
7. 섹션별 field mapping과 구현 메모를 정리한다.
8. 즉시 구현 가능한 `v1`과 향후 확장용 `v2`를 분리한다.

## 기대 산출물

최종 산출물에는 아래가 반드시 포함되어야 한다.

### 1. 샘플 형식 비교 요약

- 각 자문사/기관 샘플의 형식 특징
- 공통 섹션
- 차별점
- OPM에 참고할 점
- OPM에 맞지 않는 점

### 2. 추천 Word 양식 1안

다음을 포함해야 한다.

- 표지/헤더 구조
- 회사/주총 메타데이터 영역
- 전체 권고 요약 영역
- 안건별 권고 섹션
- 근거/증거/법률·정책 레이어 표시 방식
- caveat / review-needed / uncertain case 표시 방식
- 부록/첨부 구조

### 3. 섹션별 템플릿 스펙

각 섹션마다 다음을 써야 한다.

- 섹션 목적
- 필수 필드
- 선택 필드
- 현재 OPM 데이터로 생성 가능한지 여부
- Word 상 권장 표현 형식

### 4. OPM field mapping

현재 OPM 데이터 필드를 Word 보고서 섹션에 어떻게 매핑할지 적는다.

예:

- `meeting metadata -> header`
- `agenda recommendation -> 안건별 권고 요약`
- `evidence refs -> 근거/출처 각주 또는 부록`

### 5. 부족한 데이터 / 추가 파생 로직

다음을 명시한다.

- 현재 없는 필드
- 있으면 좋은 필드
- 현재 필드로는 표현이 애매한 항목
- 후처리/파생이 필요한 항목

### 6. 구현 메모

engineering 관점에서 다음을 적는다.

- v1에서 바로 구현 가능한 범위
- v2에서 확장할 범위
- 템플릿 엔진/Word export에서 주의할 점
- 표/머리말/각주/페이지 브레이크/부록 처리 시 유의사항

## 권장 출력 관점

최종 권고안은 아래 관점을 균형 있게 봐야 한다.

- 읽는 사람이 빠르게 결론을 파악할 수 있는가

- OPM의 실제 evidence 구조를 무리 없이 녹일 수 있는가
- 나중에 여러 스타일(`compact`, `standard`, `full`)로 확장 가능한가
